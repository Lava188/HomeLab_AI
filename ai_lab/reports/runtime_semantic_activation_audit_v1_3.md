# Runtime Semantic Activation Audit v1.3

## Executive summary

Backend runtime is loading `retriever_v1_3`, but semantic retrieval is **not active** at runtime.

The v1.3 artifact directory contains `chunk_embeddings.npy` and `faiss.index`, and `backend/.env` selects `HOMELAB_RETRIEVER_VERSION=v1_3`. However, the Node runtime loader only loads embeddings when the manifest embedding file has a `.json` extension. Because the active artifact points to `chunk_embeddings.npy`, `embeddingsByChunkId` is empty. The retriever therefore computes a local lexical bag-of-words query vector, cannot compare it to any loaded semantic vectors, assigns `semanticScore = 0` for all chunks, and reports `runtimeMode = lexical_only`.

Answer behavior is currently dominated by lexical token overlap, hardcoded source hints, query rewrite rules, topic intent rules, router keyword boosts, and policy pattern checks. The current default runtime should **not** be switched or claimed as semantic/hybrid-active yet.

### Retriever runtime evidence

`retriever.service.js` creates a query vector with `vectorizeQuery()` at line 599, but this is a local lexical vector over `runtimeCache.vocabIndex`, not an embedding model call.

Chunk semantic score is computed only if `embeddingsByChunkId[chunk.chunk_id]` is an array:

- line 610: reads `chunkVector`
- lines 610-612: `semanticScore = Array.isArray(chunkVector) ? dotProduct(queryVector, chunkVector) : 0`

Because `loadedEmbeddingVectorCount = 0`, every `chunkVector` is missing and every `semanticScore` is `0`.

Runtime mode is selected by `embeddingConfig.index_type === "python_dot_product"` at lines 680-683. The active v1.3 config is `IndexFlatIP`, so runtime reports `lexical_only`.

## Is semantic retrieval active at runtime?

**NO.**

The backend loads `retriever_v1_3` metadata and chunks, but does not load `.npy` embeddings, does not read `faiss.index`, does not call an embedding model for the query, and produces `semanticScore = 0` for all audited chunks.

## Direct answers

- Backend có đọc được `chunk_embeddings.npy` không? **Không.** File tồn tại, nhưng loader chỉ đọc `.json`; `.npy` trả `{}`.
- Backend có đọc được `faiss.index` không? **Không.** File tồn tại, nhưng runtime Node không có code đọc FAISS index.
- Runtime có tạo query embedding không? **Không theo nghĩa semantic embedding.** Nó tạo lexical bag-of-words vector bằng `vectorizeQuery()`.
- Semantic score có được tính thật không? **Không.** Code có nhánh dot product, nhưng không có chunk vectors được load nên mọi `semanticScore` bằng `0`.
- `runtimeMode` hiện tại là gì? **`lexical_only`.**
- Trường semantic trong debug/meta có bao giờ khác 0 không? **Không trong runtime hiện tại với v1_3.** Script audit ghi nhận tất cả top chunks có `semanticScore: 0`.
- Lexical/rule đang chiếm vai trò chính ở đâu? Router intent, query rewrite, topic intent, source hints, chunk scoring, topic source preference, coherence filter, and policy mode selection.

## Why current behavior is keyword-driven

1. Router intent is prototype TF-IDF plus keyword boost:
   - `INTENT_DEFINITIONS` hardcodes flow keywords and exemplars in `router-intent.service.js`.
   - `keywordHits.length * 0.08` directly boosts route score.
   - Short/low-confidence guards can block health RAG before retrieval.

2. Retriever query handling is rule-heavy:
   - `SOURCE_HINTS` maps source IDs to hardcoded keyword lists.
   - `QUERY_SYNONYM_RULES` and `COMBINATION_REWRITE_RULES` add manual expansions and forced topics.
   - `detectTopicIntent()` uses `includes()` over hardcoded topic signals.
   - `TOPIC_QUERY_EXPANSIONS` injects topic keywords such as `sepsis`, `chest pain`, and `blood test`.

3. Chunk scoring is lexical:
   - Tokens are matched against normalized title, keywords, tags, test types, and content.
   - Source hints add fixed score boosts.
   - Red-flag and test sections get additional keyword-triggered boosts.
   - Final score is nominally hybrid, but because `semanticScore = 0`, it is effectively normalized lexical score scaled by `0.55`.

4. Policy selection is pattern-driven:
   - `EMERGENCY_PATTERNS` and `TEST_QUERY_HINTS` use `includes()` checks.
   - Policy mode depends heavily on top chunk section/FAQ type and hardcoded emergency/test patterns.

5. Answer generation is extractive/template-based:
   - `answer.service.js` takes lead sentences from selected chunks and appends fixed safety text.
   - It does not repair retrieval errors with semantic generation.

## Exact blockers

1. `chunk_embeddings.npy` is not Node-readable by current loader.
   - Current loader only supports `.json` embeddings.

2. `faiss.index` is not used by backend runtime.
   - There is no FAISS binding, subprocess call, or service call in the inspected Node runtime.

3. No runtime query embedding model is active.
   - `vectorizeQuery()` builds a lexical vector over artifact vocabulary.
   - It does not use `intfloat/multilingual-e5-small` or a model server.

4. Runtime mode detection does not match current artifact config.
   - Runtime only reports hybrid when `embeddingConfig.index_type === "python_dot_product"`.
   - v1.3 uses `IndexFlatIP`, so runtime reports `lexical_only`.

5. Debug/meta omits lexical and semantic scores in API response.
   - Direct retriever output includes them, but `rag.service.js` only exposes `score` and `matchedTerms` in `meta.topChunks`.

## Recommended implementation options

### A. Node-readable embedding format

Export `chunk_embeddings.npy` to a Node-readable JSON or binary format with explicit `chunk_id -> vector` mapping, then update the loader to load that format. Also add a real query embedding provider in Node or via API/model server so `queryVector` has the same embedding space and dimension as chunk vectors.

Pros:
- Keeps one backend service.
- Simple deployment once vectors and query embeddings are available.
- Easier to inspect in Node tests.

Cons:
- JSON embeddings can be large.
- Still needs a real runtime embedding model/provider.
- Must guard vector dimension and normalization carefully.

### B. Python FAISS microservice

Run retrieval in a small Python service that loads `faiss.index`, `chunk_metadata.json`, `kb_chunks_v1_3.json`, and the same embedding model. Node calls it for top-k chunks.

Pros:
- Uses existing FAISS artifact directly.
- Aligns with current Python eval/build pipeline.
- Avoids reimplementing FAISS and `.npy` handling in Node.

Cons:
- Adds service boundary and deployment complexity.
- Requires health checks, timeout handling, and fallback behavior.

### C. Temporary hybrid bridge

Keep current lexical retriever as a fallback, but add a thin semantic bridge that shells out to a Python script or local service for v1.3 semantic top-k. Compare semantic top-k and current lexical top-k in debug before making it authoritative.

Pros:
- Fastest path to verify semantic activation without a full service migration.
- Lets the team run shadow-mode comparisons.
- Reduces risk before changing user-facing behavior.

Cons:
- Temporary architecture can become sticky.
- Shelling out per request may be slow unless cached or daemonized.
- Needs clear cutoff criteria before promoting.

## Recommendation for next step

Do **not** switch default runtime behavior now. Keep `v1_3` runtime behind the current behavior until semantic activation is real and observable.

Recommended next step: implement option **C** first as a controlled bridge or shadow-mode check, because the artifact already contains FAISS and `.npy` files and the project has Python eval scripts that know how to use them. Once semantic retrieval is validated against runtime queries, promote either option **B** for production service separation or option **A** if the team wants a pure Node runtime.

Minimum acceptance criteria before any default switch:

- Runtime loads non-empty semantic vectors or calls FAISS service successfully.
- Query embedding uses the same model/config as the artifact.
- `runtimeMode` reports semantic/hybrid truthfully.
- At least one audited query shows non-zero semantic scores.
- API/debug metadata exposes semantic score, lexical score, and final score for inspection.
- Sepsis query without literal `sepsis` routes and retrieves based on meaning, not only keyword presence.

## Confirmed no-switch decision

No backend logic, frontend code, KB files, artifacts, or default runtime settings were changed by this audit. The only added files are:

- `backend/scripts/audit_runtime_semantic_v1_3.js`
- `ai_lab/reports/runtime_semantic_activation_audit_v1_3.md`

The current runtime should be treated as **lexical/rule-based with inactive semantic scoring**, not as semantic or true hybrid retrieval.
